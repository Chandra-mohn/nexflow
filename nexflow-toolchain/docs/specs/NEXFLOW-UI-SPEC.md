## 1. Overview

The Nexflow Visual Designer is a **topology-focused canvas** for designing L1 Process pipelines. It generates `.proc` DSL files that integrate seamlessly with the existing VS Code + LSP toolchain.

### 1.2 Goals

| Goal | Description |
|------|-------------|
| **Topology Visualization** | Visual canvas for designing process data flow |
| **DSL Generation** | Generate valid `.proc` files from visual design |
| **VS Code Complement** | Enhance, not replace, the code editing experience |
| **Layer Separation** | Process canvas only; other layers have separate views |
| **Metadata Isolation** | UI-specific data stored separately from DSL |

### 1.3 Non-Goals (V1 Scope Boundaries)

| Exclusion | Rationale |
|-----------|-----------|
| âŒ Embedded code editor | VS Code + LSP already excellent |
| âŒ Monaco Editor integration | Avoid duplicating VS Code capabilities |
| âŒ L2/L3/L4 inline editing | Keep canvas focused; use layer-specific editors |
| âŒ Bidirectional live sync | Complexity; defer to V2 |
| âŒ Complex expression editing | Use VS Code for logic details |
| âŒ Multi-process canvas | One `.proc` file = one canvas |


### 2.3 Node ID Convention

Format: `{type}:{identifier}`

| Type | Example | Description |
|------|---------|-------------|
| `process` | `process:order_enrichment` | Process block |
| `stream` | `stream:raw_orders` | Kafka stream/topic |
| `xform-ref` | `xform-ref:normalize_amount` | Transform reference |
| `rules-ref` | `rules-ref:validation_rules` | Rules reference |
| `marker` | `marker:eod_complete` | EOD marker |
| `route` | `route:order_router` | Routing block |
| `join` | `join:order_customer_join` | Join block |
| `window` | `window:5min_tumbling` | Window definition |

### 2.4 Schema Versioning

- Version field enables future schema evolution
- Parser checks version and migrates if needed
- Breaking changes increment major version

### 3.2 Node Properties

#### Stream Node
| Property | Type | Description |
|----------|------|-------------|
| `name` | string | Stream identifier (Kafka topic logical name) |
| `direction` | enum | `input` \| `output` |
| `schema` | string? | Optional L2 schema reference |

#### Process Node
| Property | Type | Description |
|----------|------|-------------|
| `name` | string | Process identifier |
| `description` | string? | Optional description |
| `parallelism` | int? | Flink parallelism hint |

#### Transform Reference
| Property | Type | Description |
|----------|------|-------------|
| `xformName` | string | Reference to `.xform` file |
| `alias` | string? | Variable name for output |

#### Rules Reference
| Property | Type | Description |
|----------|------|-------------|
| `rulesName` | string | Reference to `.rules` file |
| `resultVar` | string? | Variable for result |

#### Route Node
| Property | Type | Description |
|----------|------|-------------|
| `name` | string | Router identifier |
| `conditions` | array | Routing conditions (basic) |

### 3.3 Edge Types

| Edge Type | Visual | Description |
|-----------|--------|-------------|
| **Data Flow** | Solid line | Stream data connection |
| **Signal Flow** | Dashed line | Marker/phase signals |
| **Reference** | Dotted line | Transform/Rules reference |

### 3.4 Edge Properties

| Property | Type | Description |
|----------|------|-------------|
| `variableName` | string? | Variable binding for `receive` (e.g., `order`) |
| `edgeType` | enum | `data` \| `signal` \| `reference` |
| `label` | string? | Optional display label |

**Why Not Blank?**

Many visual editors show an empty panel when nothing is selected. This wastes valuable screen space. Instead:

1. **Always useful** - Panel provides value regardless of selection state
2. **Process-level editing** - Some properties belong to the process, not individual nodes
3. **Discoverability** - Quick actions help users find common operations
4. **Context awareness** - User always sees which file they're editing

| **Export as PNG** | Creates image for documentation, presentations |

### 5.2 Auto-Layout Strategy

When `.proc.ui` doesn't exist:

1. **Parse** `.proc` â†’ extract nodes and edges
2. **Layout** using dagre.js:
   - Left-to-right flow direction
   - Input streams on left
   - Output streams on right
   - Vertical stacking for parallel branches
3. **Generate** `.proc.ui` with computed positions
4. **User Adjusts** positions as needed (persisted to `.proc.ui`)

### 5.3 Position Preservation (Future)

When `.proc` changes but `.proc.ui` exists:

| Scenario | Behavior |
|----------|----------|
| Node unchanged | Keep position from `.proc.ui` |
| Node added | Auto-layout new node only |
| Node removed | Remove from `.proc.ui` |
| Node renamed | Match by structure if possible |

### 8.4 CLI Integration (No Python Server)

The Visual Designer uses CLI-only architecture. No Python server runs - the UI spawns CLI processes as needed.

| Operation | CLI Command | Purpose |
|-----------|-------------|---------|
| **Parse to Graph** | `nexflow parse <file>.proc --format graph` | Load .proc into canvas (nodes + edges JSON) |
| **Validate** | `nexflow validate <file>.proc` | Real-time validation feedback |
| **Generate DSL** | `nexflow generate --from-ui <file>.proc.ui` | Save canvas to .proc (V2) |
| **Build** | `nexflow build` | Compile to target runtime code |

#### 8.4.1 Graph Format Output

The `--format graph` option returns UI-consumable JSON for React Flow:

```json

  "processName": "transaction_processor",
```

#### 8.4.2 Node ID Convention

Node IDs follow the pattern `{type}:{name}`:

| Type | Example ID | Description |
|------|-----------|-------------|
| `stream` | `stream:raw_orders` | Input/output stream |
| `xform-ref` | `xform-ref:normalize` | Transform reference |
| `rules-ref` | `rules-ref:approval` | Rules table reference |
| `route` | `route:inline_0` | Inline routing condition |
| `window` | `window:tumbling(5m)` | Window operation |
| `join` | `join:orders_customers` | Join operation |
| `marker` | `marker:eod_cutoff` | EOD marker |

#### 8.4.3 Rationale

| Decision | Rationale |
|----------|-----------|
| **No Python server** | Simpler deployment, no process management |
| **CLI spawning** | Consistent with existing build toolchain |
| **DSL generator in Python** | Leverages existing parser/AST infrastructure |
| **UI self-sufficient** | Canvas operations don't require backend |

## 9. User Workflows

### 9.1 Create New Process (Visual-First)

```
1. Command: "Nexflow: New Process"
2. Enter process name â†’ Creates empty canvas
3. Drag streams, add process node, connect
4. Save â†’ Generates .proc + .proc.ui
5. Open .proc in VS Code â†’ Add business logic
```

### 9.2 Edit Existing Process

```
1. Right-click .proc â†’ "Open in Visual Designer"
2. If .proc.ui exists â†’ Load positions
3. If no .proc.ui â†’ Auto-layout â†’ Generate .proc.ui
4. Edit topology in canvas
5. Save â†’ Updates both files
6. LSP validates .proc immediately
```

### 9.3 Hybrid Workflow

```
1. Design topology in Visual Designer
2. Save â†’ .proc generated
3. Open .proc in VS Code
4. Add detailed logic (expressions, conditions)
5. LSP provides completions, validation
6. Re-open in Designer â†’ Layout preserved
```
